# AGENTS_architecture.md - 通用 C++ 代码组织与架构规范

本文件供 AI 编码代理使用，用于约束 **模块划分、目录组织、依赖方向、边界设计** 与 **新增代码时的默认决策**。

目标：让 AI 在 C++ 项目中生成的代码不仅“能跑”，还具备 **清晰边界、低耦合、易维护、易扩展** 的结构。

---

## 1. 适用范围

本规范适用于绝大多数中大型 C++ 工程，尤其是：

- 桌面应用
- CLI 工具
- 服务端程序
- 长期维护的内部业务工程
- 具备多模块协作的 C++ 项目
- 需要持续演进的库或平台型工程

本规范不限定具体框架，但适用于：

- 标准 C++ 工程
- Qt / GUI 工程
- 带数据库或文件访问的业务工程
- 带设备、网络、采集、监控等模块的工程

---

## 2. 核心目标

AI 在组织代码和设计架构时，默认追求以下目标：

1. **职责清晰**：每个模块只解决一类问题
2. **边界明确**：谁依赖谁清楚可见
3. **低耦合**：避免模块互相渗透
4. **高内聚**：相关代码尽量放在一起
5. **易扩展**：新增功能时能明确落点
6. **易测试**：核心逻辑尽量不被 UI、IO、平台细节绑死
7. **易维护**：能快速定位功能、问题与依赖关系

---

## 3. 模块划分原则

### 3.1 先按职责划分模块

模块必须优先按**业务职责或技术职责**划分，而不是按文件类型堆放。

优先：

- `user/`
- `order/`
- `device/`
- `metrics/`
- `workspace/`
- `storage/`
- `ui/`
- `core/`

避免默认使用宽泛目录承接所有内容：

- `misc/`
- `helper/`
- `utils/`
- `common/`
- `base/`

这些目录只有在职责明确且内容稳定复用时才允许存在。

### 3.2 一个模块只承担一个主职责

一个模块可以包含多份文件，但应围绕**一个相对稳定的职责中心**组织。

禁止把下面这些混在同一模块中：

- UI 渲染 + 数据访问
- 业务规则 + 平台 API 细节
- 设备采集 + 展示逻辑
- 配置解析 + 核心算法 + 网络发送

### 3.3 模块粒度保持适中

避免两种极端：

- **过细**：每个小类都拆一个模块，导致 target 爆炸
- **过粗**：一个模块承载多个无关职责，导致后期失控

默认规则：

- 能围绕一个完整职责闭合的代码，优先放在一个模块内
- 只有当依赖、测试、复用、边界已经明显独立时，再拆成单独模块

---

## 4. 目录组织原则

### 4.1 目录必须反映职责边界

目录结构应帮助人快速回答：

- 这个功能在哪
- 这个模块负责什么
- 这个模块依赖什么

### 4.2 不按“头文件 / 源文件 / 工具类”做全局分类

除非项目本身就是 SDK 并有公开头目录要求，否则不要默认这样组织：

- 所有 `.h` 放一处
- 所有 `.cpp` 放一处
- 所有“工具类”扔进 `utils/`

优先把同一模块相关文件放在一起。

### 4.3 新增代码必须有明确落点

AI 新增任何类、函数、模块时，必须优先判断：

- 它属于哪个已有职责域？
- 是否已经存在合适模块可承接？
- 是否真的需要新建模块？

禁止因为“不知道放哪”就创建：

- `temp/`
- `misc/`
- `helper/`
- `new_module/`

---

## 5. 依赖方向原则

### 5.1 依赖方向必须清晰

默认依赖方向应满足：

- 表现层依赖业务层
- 业务层依赖数据访问层 / 基础设施层
- 基础设施层不反向依赖表现层
- 通用底层模块不依赖具体业务模块

### 5.2 禁止反向依赖

默认禁止：

- `storage` 依赖 `ui`
- `device` 依赖页面组件
- `core` 依赖具体窗口类
- 通用模块依赖场景模块

如果确实需要交互，应通过：

- 明确接口
- 回调
- 事件
- 依赖注入
- 适配器

而不是直接跨层互引。

### 5.3 依赖应显式而非隐式

模块依赖关系应能在以下层面看出来：

- 目录组织
- 头文件引用
- 构建配置
- 构造函数或接口定义

避免依赖“顺便可见”的全局环境。

---

## 6. 分层与职责建议

以下是默认推荐分层，不要求项目必须字面命名一致，但职责应尽量接近。

### 6.1 表现层

负责：

- UI 组件
- 页面/窗口
- 用户交互
- 输入输出适配
- 命令入口

不负责：

- 复杂业务规则
- 数据持久化细节
- 核心算法实现

### 6.2 业务层

负责：

- 用例编排
- 业务规则
- 状态流转
- 核心流程控制

不负责：

- 具体数据库访问
- 具体 UI 绘制
- 具体平台 API 细节

### 6.3 数据访问 / 基础设施层

负责：

- 文件系统
- 数据库
- 网络
- 设备 API
- 第三方服务
- OS / 平台调用

不负责：

- 页面逻辑
- 业务流程决策

---

## 7. 接口与实现边界

### 7.1 依赖抽象边界，而不是四处直接耦合实现

当一个模块需要使用另一个模块能力时，应优先依赖：

- 清晰的类接口
- 稳定的服务接口
- 明确输入输出的数据结构

不要直接跨模块访问大量内部细节。

### 7.2 接口要小而稳定

避免：

- 一个类暴露大量无关方法
- 一个头文件包含大量跨职责内容
- 一个模块对外输出过多内部细节

默认原则：

- 对外只暴露必要能力
- 不把未来可能变化的实现细节直接暴露出去

### 7.3 公共模块要克制

`common`、`shared`、`base` 这类模块只有在满足以下条件时才适合引入：

- 被多个模块真实复用
- 职责足够稳定
- 不携带具体业务语义

否则，应优先放回具体职责模块。

---

## 8. 事件、回调与模式使用原则

### 8.1 设计模式只为解决真实问题服务

允许使用：

- 工厂模式
- 策略模式
- 观察者模式
- 适配器模式
- 组合模式

但前提是：

- 真有替换需求
- 真有解耦收益
- 真能提升可维护性

禁止为了“显得专业”机械套模式。

### 8.2 事件驱动要控制边界

适合事件机制的场景：

- UI 事件
- 状态变化通知
- 异步采集结果分发
- 一对多广播

不适合事件机制的场景：

- 简单同步调用
- 明确的一对一命令链
- 本该直接返回结果的逻辑

禁止把所有模块通信都改成事件系统。

### 8.3 单例谨慎使用

单例只适用于：

- 真正全局唯一
- 生命周期清晰
- 共享访问合理

避免把单例当成“全局变量升级版”。

---

## 9. 可测试性原则

### 9.1 核心逻辑应尽量脱离 UI / IO / 平台依赖

AI 设计新代码时，应优先让：

- 业务判断
- 数据转换
- 核心流程
- 规则计算

尽量放在可独立调用的模块中。

### 9.2 不要把难测结构当正常状态

以下情况通常说明结构需要改进：

- 一个类初始化必须依赖大量外部环境
- 一个函数既处理业务又访问文件/网络/UI
- 模块只有运行整套程序才能验证

### 9.3 测试能力影响模块划分

如果一个职责天然需要被单独验证，通常也说明它值得独立成模块。

---

## 10. 重构与演进原则

### 10.1 新增功能优先落到现有清晰模块

只有在以下情况才新建模块：

- 已有模块职责不匹配
- 新能力会长期独立演进
- 依赖关系已明显独立
- 测试和复用边界已清晰

### 10.2 重构时优先改善边界，而不是只改名字

更有价值的重构包括：

- 把 UI 逻辑从业务逻辑中剥离
- 把数据访问从核心流程中剥离
- 把膨胀的万能模块拆回职责域
- 调整依赖方向，消除反向依赖

### 10.3 演进必须保留迁移路径

AI 在调整结构时，应优先：

- 渐进式移动代码
- 保持现有行为不变
- 避免一次性大拆大改
- 让模块边界逐步变清晰

---

## 11. AI 默认操作清单

AI 在新增或修改架构相关代码时，按下面顺序判断：

1. **先判断新增逻辑属于哪个职责域**
2. **优先复用已有清晰模块，不轻易新建目录**
3. **检查该逻辑是否混入了不属于本模块的职责**
4. **检查依赖方向是否正确**
5. **确认是否把 UI / IO / 平台细节泄漏进核心逻辑**
6. **确认是否引入了膨胀型 common/base/helper 模块**
7. **确认模块是否保留了合理测试边界**
8. **如需拆分，按职责拆，不按文件类型拆**

---

## 12. AI 禁止事项

AI 在组织 C++ 工程结构时，默认禁止：

- 禁止为了整齐把所有头文件统一集中存放
- 禁止默认新建 `utils` / `helpers` / `misc` 承接未知职责代码
- 禁止把 UI、业务、数据访问混在同一个模块
- 禁止让底层模块反向依赖高层模块
- 禁止把所有跨模块需求都塞进 `common` / `base`
- 禁止为简单问题强行引入复杂模式
- 禁止把同步直接调用问题过度事件化
- 禁止一次性大规模重组目录而不考虑现有结构和迁移成本

---

## 13. 给 AI 的简短执行口令

每次对话如需快速约束，可直接应用以下规则：

```text
以后所有 C++ 代码组织与架构设计遵守以下规范：
按职责拆模块，不按文件类型堆目录；
同一职责的头文件和实现尽量放在一起；
保证模块边界清晰、依赖方向单向；
UI、业务、数据访问分层清楚；
避免万能 common/utils/base 模块膨胀；
优先让核心逻辑可测试、可扩展、易维护。
```

---

## 14. 最终总结

这份规范只有三条真正核心的落地原则：

1. **按职责组织，而不是按习惯堆放**
2. **让依赖方向和模块边界保持清晰**
3. **让结构服务于维护、测试和演进**

如果没有额外要求，AI 应优先生成：

- 职责明确的模块划分
- 可读的目录边界
- 不互相污染的依赖关系
- 对未来扩展更友好的代码结构

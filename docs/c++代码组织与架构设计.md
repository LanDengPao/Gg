## 1. 为什么代码组织和架构要一起考虑

很多 C++ 项目后期变得难维护，并不只是因为代码多，而是因为一开始没有把下面两件事分清：

1. **代码应该如何按职责组织**
2. **模块之间应该如何建立依赖关系**

前者决定“文件放哪、模块怎么拆”；后者决定“谁能依赖谁、边界在哪、以后能不能改”。

如果只有目录，没有架构，项目会逐渐变成“能跑但不好改”的状态；
如果只有架构图，没有落地的目录和模块规则，设计也会迅速失效。

---

## 2. 代码组织的目标

一个好的 C++ 工程组织，至少应满足这几个目标：

- **容易定位**：功能出问题时，能快速找到对应模块
- **容易修改**：改一处不会牵动大量无关代码
- **容易扩展**：新增功能时能明确放到哪里
- **容易测试**：模块边界清晰，便于单测和集成测试
- **容易协作**：不同人修改不同模块时不互相干扰

代码组织不是为了“看起来整齐”，而是为了降低长期维护成本。

## 3. 模块化划分的基本原则

### 3.1 先按职责拆，不按文件类型拆

优先按**功能职责**划分模块，而不是先建：

- `include/`
- `src/`
- `utils/`
- `common/`
- `helpers/`

这些过于宽泛的分类本身不能表达业务边界。

更合理的做法是按真实职责拆分，例如：

- `user/`
- `order/`
- `device/`
- `metrics/`
- `workspace/`

这样看到目录时，就能知道它解决什么问题。

### 3.2 模块粒度要适中

模块太小会导致：

- target 数量过多
- 调用链变长
- 依赖关系复杂化
- 管理成本升高

模块太大会导致：

- 一个目录里塞太多职责
- 文件越来越大
- 逻辑耦合严重
- 难以测试和替换

一个比较实用的判断标准是：

> 一个模块应能围绕一个相对完整的职责闭合，而不是只因为“代码多了”就机械拆分。

### 3.3 模块边界应先于实现细节

在落代码之前，最好先回答三个问题：

1. 这个模块负责什么？
2. 这个模块不负责什么？
3. 谁可以依赖它？

如果这三个问题说不清，后面目录再整齐，代码也会逐步失控。

## 4. 文件组织建议

### 4.1 头文件与源文件分离是语言规则，不等于头文件要集中存放

C++ 中通常会把：

- 声明写在 `.h/.hpp`
- 实现写在 `.cpp`

但这不意味着所有头文件都应该被集中放到一个独立目录。

更重要的是：

- **声明和实现是否围绕同一个模块组织**
- **外部是否只看见它该看见的接口**

### 4.2 普通业务工程推荐“头源同目录”

对于 APP、工具、服务端、内部项目，推荐：

- `.h` 和 `.cpp` 放在同一模块目录
- 按模块组织，而不是按文件类型组织

例如：

```text
src/
├── user/
│   ├── user_service.cpp
│   └── user_service.h
├── order/
│   ├── order_service.cpp
│   └── order_service.h
└── device/
    ├── device_manager.cpp
    └── device_manager.h
```

这种结构更利于快速定位与维护。

### 4.3 对外库工程要区分公开头与私有实现

如果工程本质上是 SDK 或可复用组件库，则应区分：

- 公开头：给外部用户使用
- 私有头与实现：仅库内部使用

典型结构：

```text
include/mylib/
src/
```

这不是“更正式”，而是为了维护 API 边界。

## 5. 命名规范与代码风格

### 5.1 命名的核心不是形式，而是稳定和可读

无论使用：

- 驼峰命名
- 下划线命名
- PascalCase

最重要的是：

- 同一层级的命名规则一致
- 名称能反映职责
- 不使用模糊、泛化、历史遗留式命名

例如，相比下面这种名字：

- `Manager2`
- `Helper`
- `Util`
- `DataProcessorNew`

更应优先：

- `WorkspaceRepository`
- `MetricsCollector`
- `DeviceEnumerator`
- `OrderService`

### 5.2 风格统一要靠工具落地

缩进、空格、换行、括号风格等问题，不应该靠人工争论。

推荐：

- 用 `clang-format` 统一格式
- 用静态检查工具限制明显问题
- 在评审中关注结构与职责，而不是手工格式细节

## 6. 架构设计的核心原则

### 6.1 架构首先是依赖方向设计

判断一个项目架构是否健康，不要只看有没有“分层”或“模式名”，而要看：

- 依赖方向是否清晰
- 底层是否被上层反向污染
- 业务逻辑是否依赖过多 UI / IO / 框架细节

一个常见原则是：

- 高层模块依赖低层抽象
- 具体实现依赖稳定边界
- 不要让所有层互相都能直接调用

### 6.2 内聚高、耦合低

好的架构不是让模块完全没有联系，而是让：

- 模块内部围绕单一职责聚合
- 模块之间通过清晰接口协作
- 变化频率不同的代码尽量隔离

### 6.3 架构要为变化服务

架构不是为了画图，而是为了应对变化。

如果项目未来最容易变化的是：

- UI 交互
- 数据源
- 协议格式
- 业务规则
- 平台适配

那这些部分就应该尽量与核心逻辑解耦。

## 7. 常见架构模式怎么用

### 7.1 分层架构

分层是最常见、也最容易落地的方式。

典型可以分为：

- **表示层**：UI、命令行入口、接口适配
- **业务层**：业务规则、流程编排、核心逻辑
- **数据访问层**：数据库、文件、网络、设备、外部系统访问

分层的意义不在于目录名，而在于：

- 上层通过明确接口调用下层
- 下层不反向依赖上层细节

### 7.2 设计模式

设计模式本质上是解决重复设计问题的经验总结，不是必须凑齐的清单。

常见适用场景：

- **工厂模式**：隐藏对象创建差异
- **策略模式**：替换算法或规则
- **观察者模式**：事件通知
- **适配器模式**：对接旧接口或第三方接口
- **单例模式**：谨慎使用，仅限真正全局唯一且生命周期明确的对象

重点不是“用了什么模式”，而是：

- 是否真的解决了耦合问题
- 是否降低了变更成本
- 是否让代码更容易理解

### 7.3 事件驱动架构

适用于：

- 状态变化多
- 模块之间需要异步通知
- 实时性较强
- 一对多广播明显

典型场景：

- GUI 交互
- 监控系统
- 游戏逻辑
- 设备状态采集与通知

但事件驱动也容易带来：

- 调用链不直观
- 调试困难
- 事件滥用导致逻辑分散

所以要控制事件的粒度和边界，不要把一切都做成“发消息”。

## 8. 依赖管理建议

### 8.1 依赖应显式表达

一个模块依赖谁，应在构建系统和目录关系上都能看出来。

不要依赖：

- 全局头文件路径“顺便可见”
- 某个公共基类“顺便什么都能调”
- 一个万能 common 模块承载所有跨层依赖

### 8.2 避免反向依赖

高层调用低层很正常，但不要让：

- 数据访问层依赖 UI
- 基础模块依赖业务模块
- 通用模块反过来依赖具体场景模块

一旦发生这种情况，架构很快会塌成网状结构。

### 8.3 公共模块要克制

`common`、`base`、`shared` 这类模块很容易不断膨胀。

建议：

- 只放真正跨模块稳定复用的内容
- 不要因为“不知道放哪”就丢进去
- 定期清理已经演变成业务耦合点的公共代码

## 9. 测试与架构的关系

能不能方便测试，往往能反映架构是否合理。

一个模块如果很难测试，常见原因是：

- 职责过多
- 依赖太多外部环境
- 依赖方向混乱
- 状态管理不清晰

更好的做法是：

- 让核心逻辑尽量独立于 UI 和 IO
- 把数据访问、外部系统、平台相关细节隔离出去
- 让模块有明确输入输出

测试不是架构之后再补的事情，而是架构质量的验证方式。

## 10. 重构、性能优化与架构演进

### 10.1 重构是持续动作，不是大爆炸工程

项目一旦进入维护期，结构问题通常不会一次性全部解决。

更实际的做法是：

- 先识别最容易出问题的边界
- 小步调整目录和依赖
- 每次重构只解决一类问题
- 保证行为不变，再逐步改善结构

### 10.2 性能优化不能脱离结构看

很多性能问题表面是算法问题，实际是结构问题，例如：

- 无意义的层层拷贝
- 模块职责混乱导致重复计算
- 接口设计不当导致频繁 IO
- 数据结构与访问模式不匹配

因此性能优化要结合：

- 算法
- 数据结构
- 生命周期管理
- 模块协作方式

### 10.3 架构演进要保留可迁移空间

好的架构不是一开始就“永远正确”，而是能演进。

演进时要优先保留：

- 公开接口稳定性
- 模块替换能力
- 构建关系可理解性
- 旧代码向新结构迁移的路径

## 11. 常见误区

### 误区 1：目录多就代表架构好

不是。目录只是结果，架构的本质是职责和依赖关系。

### 误区 2：设计模式越多越专业

不是。模式只是工具，滥用只会增加认知负担。

### 误区 3：所有东西都抽象成公共层

不是。过早抽象通常会产生虚假的复用。

### 误区 4：重构等以后再说

很多结构问题拖久了，会让每次新增功能都越来越贵。

### 误区 5：测试只是 QA 的事

不是。测试可达性本身就是架构设计质量的一部分。

## 12. 给团队的实用建议

如果你在做一个中大型 C++ 工程，可以先从这几个问题入手：

1. 当前目录是否按职责组织？
2. 模块之间是否有明确边界？
3. 依赖方向是否清晰？
4. 是否存在过大的万能模块？
5. 核心逻辑是否被 UI / IO / 平台细节绑死？
6. 新功能进来时，团队是否知道应该放到哪里？

如果这些问题大多回答不清，说明项目需要先补组织规则，再谈架构优化。

## 14. 最后的总结

C++ 工程的代码组织与架构设计，最终都落在三件事上：

1. **按职责组织代码，而不是按习惯堆文件**
2. **让模块边界和依赖方向清晰可控**
3. **让结构能够支撑测试、扩展和长期演进**

只要这三点持续做对，项目就会更容易维护，也更容易被 AI 和人一起接手。

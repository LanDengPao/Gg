# 项目代码介绍

本文用于说明当前 `Gg` 项目的源码组织、模块职责与构建关系，帮助开发者快速理解代码入口和主要依赖关系。

- `src/core/`：核心能力、数据与设备相关逻辑
- `src/ui/`：界面组件与应用入口

## 2. 模块划分说明

### 2.1 `src/core/`

`src/core/` 是项目的核心模块，当前通过 [src/core/CMakeLists.txt](../src/core/CMakeLists.txt) 构建为静态库 `GgCore`。

它主要负责：

- 公共类型定义
- 核心业务计算
- 数据读写与工作区持久化
- 设备信息访问
- 通用线程工具

当前包含以下子域：

#### `types/`

- `AppTypes.h`

用于放置项目共享的数据结构、类型定义或通用枚举。

#### `common/`

- `LambdaThread.h`
- `LambdaThread.cpp`

用于提供通用基础能力。当前这里主要是线程封装类，属于可被多个模块复用的基础工具。

#### 根目录核心类

- `MetricsEngine.h`
- `MetricsEngine.cpp`

这是当前核心业务能力之一，负责指标计算、数据处理或图表相关的核心数据逻辑。

#### `data/`

- `WorkspaceRepository.h`
- `WorkspaceRepository.cpp`

负责工作区数据的读取、写入、保存或恢复，属于典型的数据访问层角色。

#### `device/`

- `WinDeviceInfo.h`
- `WinDeviceInfo.cpp`

负责 Windows 平台设备信息访问，属于平台/设备适配层。

---

### 2.2 `src/ui/`

`src/ui/` 通过 [src/ui/CMakeLists.txt](../src/ui/CMakeLists.txt) 构建为静态库 `GgUi`。

它主要负责：

- 界面展示组件
- 图形绘制与交互控件
- 对核心模块能力的 UI 层封装

当前主要包含：

#### 组件层

- `TrajectoryWidget.h/.cpp`
- `SparklineWidget.h/.cpp`

这两个类属于可视化控件，负责具体图表或轨迹相关展示。

#### `app/` 应用层

- `main.cpp`
- `AppController.h/.cpp`
- `Gg.h/.cpp`
- `Gg.ui`
- `Gg.qrc`

这一层负责：

- 程序入口
- 主窗口装配
- UI 与核心模块之间的调度
- Qt Designer 界面文件与资源文件接入

其中：

- `main.cpp` 是程序入口
- `Gg.h/.cpp` 应是主窗口或主应用对象
- `AppController.h/.cpp` 用于组织界面行为与核心逻辑协作
- `Gg.ui` 是 Qt 界面定义文件
- `Gg.qrc` 是 Qt 资源文件

## 3. 构建关系说明

### 3.1 根构建入口

项目根构建文件为 [CMakeLists.txt](../CMakeLists.txt)。

当前整体构建流程是：

1. 载入全局配置 [cmake/config.cmake](../cmake/config.cmake)
2. 查找 Qt Widgets 依赖
3. 构建 `GgCore`
4. 构建 `GgUi`
5. 收集 `src/ui/app/` 下的应用入口源码
6. 生成最终可执行文件 `Gg`

### 3.2 Target 关系

当前 target 依赖关系大致如下：

```text
Gg
├── GgUi
│   └── GgCore
└── Qt Widgets / Qt Core / Setupapi
```

也就是说：

- `GgCore` 是底层核心能力库
- `GgUi` 在 `GgCore` 之上构建 UI 层
- 最终 `Gg` 可执行程序负责装配应用入口代码，并链接 UI 与核心模块

## 4. 当前 CMake 配置特点

### 4.1 全局配置文件

[cmake/config.cmake](../cmake/config.cmake) 负责统一设置：

- C++23 标准
- MSVC 编译选项
- Qt 自动 `moc/uic/rcc`
- 输出目录（`bin/`、`lib/`）

这使得根 `CMakeLists.txt` 更聚焦于模块装配，而不是堆放大量编译细节。

### 4.2 当前库划分

- `GgCore`：静态库
- `GgUi`：静态库
- `Gg`：Windows GUI 可执行程序

这种划分比较符合当前项目特点：

- 核心逻辑与 UI 初步分离
- 入口代码单独保留在 `src/ui/app/`
- 后续如果继续重构，可以在不破坏入口的前提下逐步细化核心模块

### 4.3 当前构建上的一个注意点

根 [CMakeLists.txt](../CMakeLists.txt) 目前使用了：

- `file(GLOB_RECURSE APP_SOURCES CONFIGURE_DEPENDS "src/ui/app/*.cpp")`

这能减少手工维护入口源码列表，但从长期维护角度看，可读性和显式性不如手工列出源文件。

如果后续继续强化工程规范，建议逐步改为显式 `target_sources()` 列表。

## 5. 当前代码的职责分层理解

从现有结构看，可以把项目大致理解为下面三层：

### 表现层

位于：

- `src/ui/`
- `src/ui/app/`

负责：

- Qt 界面展示
- 用户交互
- 图形控件绘制
- 应用入口与窗口装配

### 业务/核心层

位于：

- `src/core/MetricsEngine.*`
- `src/core/types/`

负责：

- 核心数据处理
- 业务状态和指标计算
- 项目内部共享类型

### 数据/平台层

位于：

- `src/core/data/`
- `src/core/device/`
- `src/core/common/` 的部分通用基础设施

负责：

- 工作区数据访问
- Windows 设备信息获取
- 线程与基础能力支持

## 6. 后续维护建议

从当前结构出发，后续维护时建议遵循以下原则：

1. 新增 UI 控件优先放到 `src/ui/`
2. 新增应用装配、主窗口控制逻辑优先放到 `src/ui/app/`
3. 新增核心业务能力优先放到 `src/core/` 下按职责拆分
4. 数据访问逻辑放到 `src/core/data/`
5. 平台适配、设备访问逻辑放到 `src/core/device/`
6. 不要把与具体业务强相关的代码继续塞进 `common/`
7. 尽量保持 `ui -> core` 的单向依赖，避免核心模块反向依赖界面层

---

## 7. 总结

当前 `Gg` 项目已经形成了较清晰的模块化结构：

- `core` 负责核心能力
- `ui` 负责界面与应用层
- 根构建文件负责装配最终程序

对新加入项目的人来说，理解顺序建议是：

1. 先看 [CMakeLists.txt](../CMakeLists.txt)
2. 再看 [src/core/CMakeLists.txt](../src/core/CMakeLists.txt) 和 [src/ui/CMakeLists.txt](../src/ui/CMakeLists.txt)
3. 然后看 `src/ui/app/main.cpp` 与 `AppController`
4. 最后按 `core -> data/device/common` 的方向深入具体实现

这样会更容易建立整个项目的结构认知。

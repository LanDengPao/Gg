# Gg 开发者技术文档

## 1. 项目定位

Gg 是一个基于 **Qt Widgets + C++ + Windows SDK** 的 Windows 桌面应用，目标是做“鼠标实时数据监控 + 性能分析”工具。

当前仓库已经不是只有空壳：
- 已有主界面和多页导航
- 已接入 Windows Raw Input 鼠标输入采集
- 已有实时指标计算
- 已有测试会话持久化、历史记录、导出能力
- 已有中英文切换

但整体仍处于 **第一版可运行原型** 阶段，距离完整产品还缺少更深入的数据分析、图表能力、配置体系和更完善的测试/发布流程。

---

## 2. 技术栈与构建

### 2.1 技术栈
- **语言**：C++
- **UI 框架**：Qt 5 Widgets
- **平台相关能力**：Windows API / Raw Input / SetupAPI
- **工程体系**：Visual Studio + Qt VS Tools / QtMsBuild

### 2.2 关键依赖
- MSVC v143
- Qt kit：`5.14.2_msvc2017_64`
- Windows 桌面运行环境

### 2.3 构建方式
- Visual Studio 2022 打开 `Gg.sln`
- Debug 构建：`msbuild Gg.sln /p:Configuration=Debug /p:Platform=x64`
- Release 构建：`msbuild Gg.sln /p:Configuration=Release /p:Platform=x64`

### 2.4 当前验证方式
当前仓库没有独立测试项目，也没有 lint。主要验证方式是：
- 能成功构建
- 运行 `x64/Debug/Gg.exe`
- 手动验证界面、采样、历史记录、导出等流程

## 3. 当前工程结构（从 C++ 高级工程视角）

如果按“分层是否清晰、依赖方向是否健康、后续是否容易演进”来审视，这个项目目前已经形成了 **单体桌面应用的雏形分层**，但还没有完全从“原型式实现”进入“工程化实现”。

### 3.1 当前可识别的层次

#### 入口层
- [main.cpp](../main.cpp)

职责单一：
- 创建 `QApplication`
- 构造主窗口 `Gg`
- 进入 Qt 事件循环

这一层是健康的，足够薄，没有把业务初始化塞进入口。

#### 表现层（Presentation / Shell）
- [Gg.h](../Gg.h)
- [Gg.cpp](../Gg.cpp)
- [SparklineWidget.h](../SparklineWidget.h)
- [SparklineWidget.cpp](../SparklineWidget.cpp)
- [TrajectoryWidget.h](../TrajectoryWidget.h)
- [TrajectoryWidget.cpp](../TrajectoryWidget.cpp)

这一层现在承担了三类职责：
1. 页面和控件构造
2. Windows 消息入口适配
3. 业务状态到界面的投影

从原型角度这是可接受的，但从工程角度看，`Gg` 已经明显是一个 **过重的 God Object 候选**：
- 既是 `QMainWindow`
- 又负责 Raw Input 接入
- 又负责页面组织
- 又负责刷新策略
- 又负责部分交互编排

`SparklineWidget` 和 `TrajectoryWidget` 则是比较健康的表现层组件：职责单一、接口简单、复用边界清晰。

#### 应用层（Application / Orchestration）
- [AppController.h](../AppController.h)
- [AppController.cpp](../AppController.cpp)

`AppController` 是当前最接近“应用服务层”的类。它负责：
- 驱动测试生命周期
- 协调指标引擎和仓储
- 对外暴露快照、历史、导出接口

从分层角度看，这一层是必要且正确的，因为它挡住了 UI 直接碰持久化和统计细节。

但它现在仍然带有一部分基础设施细节，例如：
- 直接管理 `QFile`
- 直接写 `QDataStream`
- 直接知道 `samples.bin` 的写入格式

也就是说，`AppController` 现在既像应用服务，也像一部分用例实现加一部分基础设施代码的混合体。

#### 领域/计算层（Domain / Analytics Core）
- [MetricsEngine.h](../MetricsEngine.h)
- [MetricsEngine.cpp](../MetricsEngine.cpp)
- [AppTypes.h](../AppTypes.h)

这是当前最像“核心业务”的部分。

`MetricsEngine` 的优点：
- 不依赖 QWidget
- 不直接依赖文件系统
- 以输入样本驱动状态更新
- 可以生成会话汇总

这意味着它已经具备了进一步抽成可测试核心模块的条件。

但 [AppTypes.h](../AppTypes.h) 现在是一个明显的 **大一统头文件**：
- 放了领域模型
- 放了展示层文案函数
- 放了 JSON 转换
- 放了状态文本映射

从 C++ 工程结构看，这会导致：
- 依赖扩散
- 编译边界不清
- “领域模型”和“表示层/存储层”耦合在一起

#### 基础设施层（Infrastructure）
- [WorkspaceRepository.h](../WorkspaceRepository.h)
- [WorkspaceRepository.cpp](../WorkspaceRepository.cpp)
- [WinDeviceInfo.h](../WinDeviceInfo.h)
- [WinDeviceInfo.cpp](../WinDeviceInfo.cpp)
- [LambdaThread.h](../LambdaThread.h)
- [LambdaThread.cpp](../LambdaThread.cpp)

这部分承担：
- 文件系统工作区管理
- JSON / 二进制文件读写
- Windows 鼠标设备枚举与友好名解析
- 线程工具封装

这是典型的基础设施代码，方向上应该被应用层调用，而不是反向侵入核心计算层。当前大体满足这一点。

### 3.2 当前依赖方向

如果把当前项目抽象成依赖图，大致是：

```text
main
  -> Gg
      -> AppController
          -> MetricsEngine
          -> WorkspaceRepository
      -> WinDeviceInfo
      -> 自定义 Widget
```

这个依赖方向总体是顺的，但有两个结构性问题：

#### 问题 1：UI 层知道太多
[Gg.cpp](../Gg.cpp) 不只是“显示”，它还知道：
- Windows Raw Input 细节
- `DeviceInfo` 解析流程
- 采样构造逻辑
- 刷新节奏
- 页面内部状态同步细节

这说明表现层已经开始侵入输入适配层和部分应用编排层。

#### 问题 2：共享模型层混合了多种关注点
[AppTypes.h](../AppTypes.h) 同时服务于：
- 领域计算
- UI 展示
- JSON 序列化
- 文案国际化

这在小项目里很方便，但会在后续迭代里让“改一个字段，半个工程重编译”。

---

### 3.3 从高级工程师视角的模块评价

#### `Gg`
定位应该是 **应用外壳 + 页面装配器**，但当前实现已经超出这个边界。

建议未来把它拆成至少三类对象：
- 主窗口壳层：菜单、导航、页面装配
- 输入适配器：Raw Input 注册与消息解包
- 页面/面板组件：Dashboard/Test/History/Compare/Device

#### `AppController`
当前定位合理，但后续应该更纯粹地成为“用例编排层”。

理想状态下它只做：
- start/stop test
- ingest sample
- query snapshot/history
- export session

而不是直接关心二进制流头、文件打开关闭等低层细节。

#### `MetricsEngine`
这是当前最值得保护的核心模块。

后续应该继续保持：
- 不依赖 UI
- 不依赖 Windows API
- 不依赖具体存储介质

如果项目要长期演进，测试应优先补在这里。

#### `WorkspaceRepository`
现在是“文件仓储 + 导出器”的合体。

短期可接受，长期建议拆分为：
- SessionRepository：面向会话元数据
- SampleStore：面向原始样本流
- ExportService：面向外部导出格式

#### `WinDeviceInfo`
这是清晰的平台适配代码，边界比较健康。后续只要避免它被 UI 到处直接调用即可。

#### `LambdaThread`
当前还没有进入主干流程，因此它更像“可用工具”，还不是架构中的稳定部件。没有明确异步场景前，不要围绕它提前设计太多抽象。

## 4. 当前架构应如何理解

从高级工程角度，这个项目现在最准确的描述不是“分层完善”，而是：

**一个以 Qt 主窗口为壳、以控制器为编排、以指标引擎为核心、以文件系统为存储后端的单体桌面应用。**

### 4.1 主链路

当前主流程可以概括为：

1. 应用启动，构造 [Gg](../Gg.h)
2. `Gg` 构造 [AppController](../AppController.h)
3. 窗口显示时注册 Raw Input
4. `WM_INPUT` 到达后由 [Gg.cpp:1130](../Gg.cpp#L1130) 处理
5. UI 层把原始输入转换为 `MouseSample`
6. `AppController` 接收样本并驱动 `MetricsEngine`
7. 若处于录制态，则同步写入 `samples.bin`
8. UI 通过定时刷新拉取 `LiveSnapshot`
9. 停止测试时生成 `SessionSummary` 并持久化

### 4.2 这套架构的优点
- 主链路短，容易调试
- 模块数量少，当前迭代成本低
- 核心数据流清晰：采样 -> 计算 -> 落盘 -> 展示
- 没有过度抽象，适合原型期快速推进

### 4.3 这套架构的代价
- `Gg.cpp` 过大后会迅速失控
- 输入接入、业务编排、页面构建混在同一个编译单元
- 很多逻辑仍以“定时刷新 + 主线程协作”为核心
- 单元测试切入点有限，尤其 UI 与输入部分

所以当前结构的判断应该是：

**方向是对的，但边界还偏粗，离可长期维护的桌面工程还有一次明显的结构整理。**

## 6. 输入采集与设备识别

### 6.1 Raw Input 注册
见 [Gg.cpp:637](../Gg.cpp#L637)。

当前通过：
- `usUsagePage = 0x01`
- `usUsage = 0x02`
- `dwFlags = RIDEV_INPUTSINK`

注册鼠标 Raw Input，这意味着窗口即使不在前台也能接收输入数据，用于后台录制。

### 6.2 消息处理
见 [Gg.cpp:1130](../Gg.cpp#L1130)。

主窗口覆写 `nativeEvent()`：
- 只处理 `windows_generic_MSG`
- 遇到 `WM_INPUT` 时调用 `handleRawInput()`

### 6.3 样本结构
鼠标输入会转换成 `MouseSample`：
- `timestampUs`
- 屏幕坐标 `position`
- 相对位移 `delta`
- 滚轮值 `wheelDelta`
- 按钮标记 `buttonFlags`
- 简化事件类型 `eventType`

定义见 [AppTypes.h:46-54](../AppTypes.h#L46-L54)。

### 6.4 设备信息
设备信息最终写入 `DeviceInfo`，包含：
- `deviceId`
- `displayName`
- `vendorId` / `productId`
- `manufacturer`
- `connectionType`
- `rawDeviceName`
- `connected`
- `lastSeenUtc`

主窗口先从 Raw Input 路径解析基础信息，再调用 `populateFriendlyMouseName()` 尝试补充友好名称。

## 7. 指标计算逻辑

指标引擎核心在 [MetricsEngine.cpp](../MetricsEngine.cpp)。

### 7.1 实时窗口
当前维护几个时间/容量窗口：
- 1 秒平均 Hz
- 5 秒平均 Hz
- 30 秒平均 Hz
- 最多 240 个历史点用于折线图
- 最多 320 个轨迹点用于轨迹预览
- 最多 64 个 recent delta 用于抖动估算

### 7.2 当前 Hz
相邻两个样本的时间差 `dtUs` 大于 0 时：
- `hz = 1000000 / dtUs`

### 7.3 速度
速度基于本次位移长度和相邻采样时间差估算：
- `speed = distance / dt`

单位表现为 `px/s`。

### 7.4 抖动
当前实现里的 jitter 是基于最近 delta 序列相邻差异的均值估算，不是频域上的高阶抖动分析。它更接近“短窗口运动不稳定程度”的启发式指标。

### 7.5 稳定度评分
稳定度基于平均 Hz 和标准差计算，公式见 [MetricsEngine.cpp:263-269](../MetricsEngine.cpp#L263-L269)。

现阶段它是经验公式：
- 平均值越高且标准差越小，稳定度越高
- 分数被限制在 0~100

### 7.6 会话汇总
停止测试时，由 `buildSessionSummary()` 生成：
- avg/min/max/stddev Hz
- stabilityScore
- jitterValue
- smoothnessScore
- peak/avg speed
- totalDistance
- 点击/滚轮计数
- 设备信息

## 8. 会话与存储设计

### 8.1 工作区目录
工作区根目录由 [WorkspaceRepository.cpp:47-52](../WorkspaceRepository.cpp#L47-L52) 决定：
- 默认使用 `QStandardPaths::AppDataLocation`
- 在其下创建 `workspace`

即运行时数据不写入仓库本身。

### 8.2 目录结构
每个会话目录位于：

```text
workspace/
  sessions/
    <session_id>/
      session.json
      summary.json
      samples.bin
```

### 8.3 文件说明

#### session.json
保存会话基础信息：
- sessionId
- mode
- status
- start/end time
- duration
- sample count
- sessionDir

#### summary.json
保存测试汇总统计信息。

#### samples.bin
保存原始采样流，使用 `QDataStream` 小端序写入。

文件头：
- magic: `0x47475331`
- version: `1`

之后每条记录依次写入：
- timestampUs
- x
- y
- dx
- dy
- wheelDelta
- buttonFlags
- eventType

### 8.4 导出
当前支持两种导出：
- CSV
- JSON

入口在 [Gg.cpp:1075](../Gg.cpp#L1075)，底层实现见 [WorkspaceRepository.cpp:222](../WorkspaceRepository.cpp#L222)。

注意：当前导出的是 **样本级原始数据**，不是完整报表。

## 9. 多语言机制

当前没有使用 Qt Translator，而是采用轻量级自定义双语方案：
- `UiLanguage` 枚举定义语言状态
- `setAppUiLanguage()` 设置全局语言
- `uiText(english, chinese)` 在运行时返回对应文案

相关定义见 [AppTypes.h:125-145](../AppTypes.h#L125-L145)。

优点：
- 简单直接
- 适合当前项目体量

限制：
- 文案分散在代码里
- 难以规模化管理
- 不适合未来更多语言扩展

如果后续文案数量明显增长，建议迁移到 Qt 的 `tr()` / `.ts` / Linguist 方案。

## 10. 工程结构评估与演进建议

### 10.1 当前结构的优点
- 已经形成了基本可辨识的分层：表现层 / 应用层 / 核心计算层 / 基础设施层
- 核心数据流清晰，没有过度抽象
- `MetricsEngine` 具备演进为稳定核心模块的潜力
- Windows 平台相关代码基本集中在 [Gg.cpp](../Gg.cpp) 和 [WinDeviceInfo.cpp](../WinDeviceInfo.cpp)
- 工作区存储与导出逻辑已经从 UI 中部分抽离

### 10.2 当前结构的主要问题
- [Gg.cpp](../Gg.cpp) 同时承载页面构造、消息接入、状态刷新、部分交互编排，职责过重
- [AppTypes.h](../AppTypes.h) 混合模型、展示文案、JSON 转换，关注点不纯
- [AppController.cpp](../AppController.cpp) 中仍有低层文件流细节，应用层与基础设施层边界不够干净
- 页面目前还是“主窗口内部的一组构造函数”，还不是独立组件
- 缺少围绕核心计算层的自动化测试

### 10.3 推荐的整理顺序

#### 第一阶段：先拆 UI，不改业务语义
目标：控制 `Gg` 膨胀。

优先拆分：
- DashboardPage
- TestPage
- HistoryPage
- ComparePage
- DevicePage

这样做的收益最大，因为它几乎不改变业务模型，只是把 UI 结构从一个超大 cpp 拆成多个稳定部件。

#### 第二阶段：抽输入适配层
把 Raw Input 相关逻辑从 [Gg.cpp](../Gg.cpp) 拆出成独立对象，例如：
- `RawInputMouseSource`
- 或 `MouseInputAdapter`

它负责：
- 注册 Raw Input
- 处理 `WM_INPUT`
- 产出 `MouseSample` 与 `DeviceInfo`

主窗口只负责接线，而不是自己解析输入。

#### 第三阶段：净化共享类型层
把 [AppTypes.h](../AppTypes.h) 逐步拆成：
- `DomainTypes`：纯模型
- `UiText` / `Localization`：显示文案
- `JsonConverters`：序列化转换

这是典型的“先不影响功能，但显著改善边界”的工程整理动作。

#### 第四阶段：细化持久化职责
把当前 `WorkspaceRepository` 的多职责拆开。至少在逻辑上区分：
- 会话元数据存储
- 样本流存储
- 导出服务

这一步对未来接 SQLite、批量分析、索引检索都更友好。

## 11. 建议的目录重组方案

当前仓库还是“文件平铺型”结构。对原型阶段没问题，但随着页面、输入适配、分析能力继续增长，目录会很快失去语义。

从长期维护角度，建议逐步整理为下面这种结构：

```text
src/
  app/
    AppController.h/.cpp
    usecases/
      StartTest.h/.cpp
      StopTest.h/.cpp
      ExportSession.h/.cpp
  domain/
    model/
      DeviceInfo.h
      MouseSample.h
      SessionRecord.h
      LiveSnapshot.h
    metrics/
      MetricsEngine.h/.cpp
    services/
      SessionAnalyzer.h/.cpp
  infra/
    input/
      RawInputMouseSource.h/.cpp
      WinDeviceInfo.h/.cpp
    storage/
      WorkspaceRepository.h/.cpp
      SessionRepository.h/.cpp
      SampleStore.h/.cpp
    export/
      SessionExportService.h/.cpp
    threading/
      LambdaThread.h/.cpp
  ui/
    mainwindow/
      MainWindow.h/.cpp
    pages/
      DashboardPage.h/.cpp
      TestPage.h/.cpp
      HistoryPage.h/.cpp
      ComparePage.h/.cpp
      DevicePage.h/.cpp
    widgets/
      SparklineWidget.h/.cpp
      TrajectoryWidget.h/.cpp
    localization/
      UiText.h/.cpp
main.cpp
```

### 11.1 这个结构的核心价值
- `app` 负责用例编排，不关心 QWidget 细节
- `domain` 放核心模型和算法，尽量纯
- `infra` 放 Windows / 文件系统 / 导出 / 线程等外部依赖
- `ui` 只做展示、交互和装配

这套结构不是为了“分层好看”，而是为了让依赖方向稳定：

**ui -> app -> domain**  
**ui/app -> infra**

而不是让 `domain` 反向依赖 UI 或平台实现。

### 11.2 结合当前项目的最低成本落地方式
不建议一次性大搬家。更适合的顺序是：

1. 先建 `ui/pages`，把 5 个页面拆出去

3. 再把 [AppTypes.h](../AppTypes.h) 拆成多个小头文件
4. 最后再细化 `storage/export` 目录

也就是说，先做“高收益、低语义风险”的整理，再做真正的架构细化。

## 12. 建议的类拆分与职责边界

当前最需要拆的不是算法，而是 [Gg.cpp](../Gg.cpp)。

### 12.1 主窗口建议拆分结果

#### `MainWindow`
建议保留的职责：
- 应用级菜单与导航
- 页面装配
- 控制页面切换
- 统一接线 controller 与各 page

不建议继续放在这里的职责：
- Raw Input 解包
- 设备路径解析
- 各页面具体控件刷新细节
- 大量页面内文案/布局代码

#### `DashboardPage`
职责：
- 展示实时快照
- 渲染统计卡片、火花线、轨迹预览
- 提供“最近一次测试”入口动作

接口建议：
- `setSnapshot(const LiveSnapshot&)`
- `setLastSession(const SessionRecord&)`

#### `TestPage`
职责：
- 展示测试模式选择
- Start / Stop 交互
- 展示录制态指标和结束结果

接口建议：
- `setSnapshot(const LiveSnapshot&)`
- `showLastResult(const SessionRecord&)`
- 发出 `startRequested(TestMode)` / `stopRequested()` 信号

#### `HistoryPage`
职责：
- 展示历史表格
- 管理当前选中项
- 发出导出/加入对比请求

接口建议：
- `setHistory(const QVector<SessionRecord>&)`
- `selectedSessionIds() const`

#### `ComparePage`
职责：
- 接收选中的会话集合
- 展示对比结果
- 提供清空操作

#### `DevicePage`
职责：
- 展示当前设备元数据
- 不参与设备发现逻辑

### 12.2 输入适配层建议

#### `RawInputMouseSource`
建议新建的职责：
- 注册/注销 Raw Input
- 接收并解析 `WM_INPUT`
- 维护设备句柄到 `DeviceInfo` 的缓存
- 向外发出标准化样本事件

理想接口：
- `bool start(HWND hwnd)`
- `bool handleNativeMessage(MSG* msg)`
- signal/callback：`sampleReady(const MouseSample&, const DeviceInfo&)`

这样一来，主窗口只负责把 native message 转交给它，而不是自己解析整个输入协议。

### 12.3 应用层建议

#### `AppController`
建议收敛为下面四类接口：
- 命令：`startTest()` / `stopTest()` / `ingestSample()`
- 查询：`snapshot()` / `history()` / `lastSession()`
- 导出：`exportSession()`
- 同步状态：`setUiActive()` / `reloadHistory()`

低层细节建议逐步移出：
- `QFile`
- `QDataStream`
- 样本文件头写入
- 文件打开关闭

这些更适合进入 `SampleStore` 或类似基础设施类中。

### 12.4 共享模型的拆分建议

当前 [AppTypes.h](../AppTypes.h) 太重，建议拆成：
- `DomainEnums.h`：`TestMode`、`SessionStatus`
- `DeviceInfo.h`
- `MouseSample.h`
- `SessionTypes.h`：`SessionSummary`、`SessionRecord`
- `LiveSnapshot.h`
- `UiText.h`：语言切换与显示文本
- `JsonConverters.h/.cpp`：`toJson` / `fromJson`

这是一个非常典型的整理动作，收益在于：
- 减少头文件污染
- 缩小编译影响面
- 降低 UI / domain / storage 的耦合

### 12.5 持久化类的拆分建议

#### `SessionRepository`
职责：
- 保存/读取 `session.json`
- 保存/读取 `summary.json`
- 加载历史会话清单

#### `SampleStore`
职责：
- 写入 `samples.bin`
- 读取二进制样本流
- 管理文件头版本

#### `SessionExportService`
职责：
- 导出 CSV
- 导出 JSON
- 后续扩展完整报告导出

这样可以避免一个 `WorkspaceRepository` 同时承担：目录管理、元数据仓储、样本存储、导出器 四种角色。

## 13. 可执行的重构路线图

这里不谈原则，直接给可以执行的顺序。

### 13.1 第 0 步：先立边界，不先搬文件
先约束新代码写法：
- 不再往 [Gg.cpp](../Gg.cpp) 追加新的页面构造函数
- 不再往 [AppTypes.h](../AppTypes.h) 继续塞 UI 文案和更多转换函数
- 新增功能优先落到独立 page / adapter / service 中

这一步几乎没有成本，但能先止血。

### 13.2 第 1 步：拆页面，保持行为不变
目标：把 [Gg.cpp](../Gg.cpp) 从“超大实现文件”降成“装配文件”。

建议顺序：
1. `HistoryPage`
2. `ComparePage`
3. `DevicePage`
4. `TestPage`
5. `DashboardPage`

理由：
- `History/Compare/Device` 交互相对简单，适合作为第一批拆分对象
- `Dashboard/Test` 与实时刷新耦合更深，放后面更稳

完成标准：
- `Gg` 不再持有每个页面内部的全部控件成员
- 页面刷新通过 `setXxx(...)` 接口完成
- 页面行为保持不变

### 13.3 第 2 步：抽 Raw Input 适配器
目标：把平台输入从 UI 壳里拿出来。

建议新增：
- `RawInputMouseSource.h/.cpp`

迁移内容：
- `registerRawInput()`
- `handleRawInput()`
- `resolveDeviceInfo()`
- `m_deviceCache`
- 与 `WM_INPUT` 解析直接相关的逻辑

保留在主窗口中的内容：
- `nativeEvent()` 只做转发
- 收到标准化样本后调用 controller

完成标准：
- 主窗口不再知道 Raw Input 数据结构细节
- Windows 输入协议只集中在一个模块中

### 13.4 第 3 步：拆共享类型
目标：降低头文件污染和编译耦合。

建议先拆：
1. `UiText` 相关函数
2. JSON 转换函数
3. domain model 结构体

推荐顺序：
- 先把 `uiText()` 和语言相关状态移出
- 再把 `toJson()/fromJson()` 移出
- 最后拆分模型头文件

原因：
- 这样最不容易引发大面积 include 断裂
- 每一步都可以单独编译验证

### 13.5 第 4 步：拆存储职责
目标：让应用层不再碰底层样本流文件细节。

建议拆出：
- `SessionRepository`
- `SampleStore`
- `SessionExportService`

迁移顺序：
1. 先拆 `SampleStore`，把 `QDataStream` 与 magic/version 收进去
2. 再拆 `SessionExportService`
3. 最后视情况把 `WorkspaceRepository` 缩成目录/路径协调者，或直接让它消失

### 13.6 第 5 步：补测试
优先级最高的测试对象不是 UI，而是：
- `MetricsEngine`
- JSON 转换
- 样本文件读写

建议最先覆盖：
- 连续采样下 Hz 计算
- session summary 结果
- `samples.bin` 的版本头与读写一致性

## 14. 工程整理原则与评审检查清单

这部分适合以后做代码评审时直接对照。

### 14.1 工程整理原则

#### 原则 1：先拆边界，再谈抽象
优先把职责从大类里搬出去，而不是先设计一堆接口。

#### 原则 2：先保持行为一致，再优化实现
结构整理阶段不要顺手改业务逻辑、指标公式或 UI 行为。

#### 原则 3：平台代码集中，业务代码纯化
Windows API、SetupAPI、Raw Input 这类代码应该尽量收拢到 `infra/input`。

#### 原则 4：共享类型不要混入展示与存储细节
domain model 应该尽量纯，展示文本和 JSON 转换要分离。

#### 原则 5：页面只负责显示与交互，不负责数据来源
页面可以发信号，不应该自己打开文件、查仓储、做系统调用。

### 14.2 代码评审检查清单

#### UI 相关检查
- 是否又把大量控件逻辑塞回了 [Gg.cpp](../Gg.cpp)？
- 页面是否直接调用仓储或 Windows API？
- 页面刷新是否通过明确接口完成，而不是外部逐个控件赋值？

#### 输入适配检查
- Raw Input 解析是否集中在单一模块？
- 是否把 Windows 结构体细节泄漏到了页面层？
- 设备缓存是否只在输入适配层维护？

#### 应用层检查
- `AppController` 是否只做编排，而不是亲自读写底层文件？
- 新增操作是否有清晰命令/查询边界？

#### 领域层检查
- `MetricsEngine` 是否仍然独立于 UI 和平台 API？
- 新增指标是否进入了核心计算层，而不是散落在界面代码里？

#### 存储层检查
- 二进制样本格式是否有明确版本控制？
- JSON 字段变更是否考虑旧数据兼容？
- 导出逻辑是否与仓储逻辑混杂？

### 14.3 判断一次重构是否成功的标准
- [Gg.cpp](../Gg.cpp) 明显缩小
- 新增页面不需要再改主窗口大量成员定义
- 输入协议细节不再出现在 UI 页面中
- `AppTypes.h` 不再继续膨胀
- `AppController` 不再直接管理样本文件流

## 18. 后续开发建议

建议优先改这些位置：
1. 领域模型定义

- 页面独立成类

- 页面间跳转由 MainWindow 或 Controller 统一协调

建议优先用于这些场景：
- 大批量历史重算
- 报表导出
- 更复杂的轨迹分析
- 设备扫描/刷新

但要先定义任务边界，再决定是否复用 [LambdaThread.h](../LambdaThread.h)。不要因为有线程工具就提前并发化。

### 18.4 做历史数据增强时
如果后续需要：
- 按设备筛选
- 按时间检索
- 多维统计报表
- 历史趋势分析

建议考虑将文件存储升级为：
- SQLite 元数据索引 + 原始样本文件
或
- SQLite 统一存储

## 19. 开发时的注意事项

### 19.1 改动 Raw Input 相关逻辑时
要重点关注：
- 是否误吞其他系统消息
- 后台采集是否仍然可用
- 多鼠标设备时的设备识别是否退化

### 19.3 改动导出时
要确认：
- CSV 列顺序是否稳定
- JSON 字段命名是否一致
- 是否需要同时导出 summary

### 19.4 改动语言切换时
要确认：
- `retranslateUi()` 是否覆盖到了新增控件
- 表头、按钮、状态栏文案是否都同步刷新

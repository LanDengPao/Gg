# 鼠标实时数据 PC App 技术方案草案

## 1. 总体架构
建议采用“采集层 / 计算层 / 存储层 / 展示层”四层结构：

- 采集层：负责从 Windows 原始输入接口持续接收鼠标事件
- 计算层：负责轮询率、稳定度、轨迹抖动、统计汇总等指标计算
- 存储层：负责将会话、原始采样数据、统计结果、设备信息写入工作目录文件库
- 展示层：负责 Qt UI 页面、图表、状态面板、历史对比展示

核心原则：**采集与录制不能依赖 UI，展示层只消费数据，不驱动数据产生**。

## 2. 线程与模块划分建议
建议至少拆分以下职责：

1. UI 主线程
- 负责窗口、页面切换、图表刷新、用户操作响应
- 不直接承担高频采样计算与磁盘写入

2. 输入采集线程
- 负责接收 Raw Input / Windows 鼠标输入事件
- 将事件标准化为统一采样结构
- 推送到内存缓冲区或事件队列

3. 统计计算线程
- 消费采样事件
- 计算实时指标、窗口统计值、评分结果
- 生成供 UI 使用的轻量快照数据

4. 录制/存储线程
- 将原始采样数据与测试会话数据异步写入工作目录文件库
- 负责顺序追加写入、刷盘节流、会话结束汇总

可复用现有 [LambdaThread.h](LambdaThread.h) 作为后台线程承载基础，但后续建议逐步明确独立模块边界。

## 3. 数据流设计
建议数据流如下：

1. Windows 输入事件进入采集层
2. 采集层生成标准化 MouseSample
3. MouseSample 同时进入两条链路：
   - 实时计算链路：用于当前轮询率、轨迹、抖动、状态摘要
   - 录制链路：用于原始数据持久化
4. 计算层按固定频率输出 UiSnapshot
5. UI 活跃时消费 UiSnapshot 并刷新图表
6. UI 最小化或不活跃时，暂停高频图表刷新，但采集链路和录制链路继续运行
7. 测试结束后生成 SessionSummary，用于历史记录与对比页面

## 4. 后台录制与前台可视化策略
建议明确区分两套状态：

### 4.1 应用可见性状态
- 前台活跃
- 最小化
- 后台非活跃

### 4.2 业务运行状态
- 空闲
- 实时监控中
- 测试录制中
- 测试结束待查看

处理原则：
- 只要业务状态处于“测试录制中”，即使窗口最小化，也必须持续采集与写盘
- 前台活跃时，允许高频刷新图表和实时数值
- 最小化或后台非活跃时，暂停非必要图表刷新，只保留轻量状态更新
- 恢复前台后，UI 从最新快照恢复，不要求重放全部历史帧

## 5. 内存缓冲与节流建议
为避免高频鼠标事件直接压垮 UI 或存储层，建议：

- 采集层使用环形缓冲区或无锁/低锁队列暂存事件
- 存储层采用批量写入，不对每个事件单独写一次
- UI 刷新频率与采样频率解耦，例如 UI 每秒刷新固定次数
- 图表展示使用时间窗口数据，而非无限累积点数
- 长时间录制时，对内存中的可视化缓存做滚动裁剪

## 7. 文件库结构草案

### 7.1 设计原则
- 原始采样与汇总结果分离存储
- 高频原始数据采用顺序追加文件写入
- 会话元信息与统计结果使用易读文本文件保存
- 设备信息单独归档，避免每条采样重复写入大量冗余字段
- 第一版优先保证可录制、可查询、可导出，不引入数据库依赖

### 7.2 工作目录结构建议
建议工作目录按如下组织：

- workspace/
  - manifest.json
  - devices/
    - devices.json
  - sessions/
    - <session_uuid>/
      - session.json
      - summary.json
      - samples.bin
      - samples.index.json（可选）
      - events.csv（导出时生成，可选）

说明：
- `manifest.json`：记录文件库版本、应用版本、最近访问记录、全局配置
- `devices.json`：记录设备缓存信息
- `session.json`：记录会话元信息
- `summary.json`：记录会话汇总指标
- `samples.bin`：记录原始高频采样
- `samples.index.json`：可选，用于快速定位、分段信息或辅助回放
- `events.csv`：不是主存储，仅在导出时生成

### 7.3 查询场景建议
- 首页摘要：读取当前内存状态 + 最近一个 session 的 `summary.json`
- 历史记录页：扫描 `sessions/` 下各会话目录并读取 `session.json`
- 对比页：批量读取多个会话目录下的 `summary.json`
- 原始数据导出：读取 `samples.bin` 转换为 CSV / JSON
- 设备信息页：读取 `devices.json`

### 7.4 数据归档与容量建议
- 长时间录制时，主要容量来自 `samples.bin`
- 第一版先按会话拆目录，天然便于删除、备份和归档
- 导出完成后允许用户手动删除整个会话目录
- 后续如需超长时间后台录制，可考虑 `samples.bin` 分段滚动写入

## 8. 文件格式定义 V0.1

### 8.1 通用约定
- 文本文件统一使用 UTF-8 编码
- JSON 文件统一使用小写 snake_case 字段命名
- 时间字段统一使用 ISO 8601 字符串，UTC 保存
- 布尔值统一使用 true / false
- 所有格式文件都需要版本字段，便于后续兼容
- 未识别或暂时不可用的字段允许缺省或为 null，UI 显示为 N/A

### 8.2 manifest.json
用途：描述整个工作目录文件库的全局信息。

建议字段：
- format_version
- app_version
- workspace_id
- workspace_name
- created_at
- updated_at
- last_opened_at
- current_session_uuid
- recent_session_uuids
- device_registry_path
- sessions_root
- settings

### 8.3 session.json

- sample_count

- created_at
- updated_at

### 8.4 summary.json
用途：保存单次会话的汇总指标和可直接用于历史/对比页的数据。

建议字段：

- metric_version

- min_hz

- stability_score

### 8.5 devices.json
用途：保存工作目录中识别过的设备信息缓存。

建议结构：
- format_version
- updated_at
- devices: []

单个 device 建议字段：
- device_id
- vendor_id
- product_id
- manufacturer
- product_name
- serial_number
- firmware_version
- connection_type
- battery_level
- first_seen_at
- last_seen_at
- extra_info

### 8.6 samples.bin
用途：保存高频原始采样数据，要求顺序追加写入、高效读取。

建议文件结构：
1. 文件头（Header）
2. 固定长度记录区（Records）

#### Header 建议字段
- magic: 8 bytes
- format_version: uint16
- header_size: uint16
- record_size: uint16
- reserved: uint16
- session_uuid: fixed bytes
- start_time_unix_us: uint64
- device_id_hash: uint64

#### 单条记录建议字段
- timestamp_us: uint64
- event_index: uint32
- pos_x: int32
- pos_y: int32
- delta_x: int16 或 int32
- delta_y: int16 或 int32
- wheel_delta: int16
- button_flags: uint16
- event_type: uint8
- reserved: 若干补齐字节

说明：
- 第一版建议使用固定长度记录，避免变长结构增加解析复杂度
- 如果担心 delta 溢出，统一使用 int32 更稳妥
- `event_type` 建议枚举：0=move, 1=button, 2=wheel, 3=combined

### 8.7 samples.index.json（可选）
用途：辅助快速定位或分段统计，不作为第一版强依赖。

建议字段：
- format_version
- session_uuid
- sample_count
- chunk_count
- chunks

### 8.8 版本兼容策略
- `manifest.json`、`session.json`、`summary.json`、`devices.json` 都必须带 `format_version`
- `samples.bin` 通过 header 中的 `format_version` 和 `record_size` 做兼容
- 新增字段时优先保持向后兼容，不要轻易改写旧字段语义
- 升级格式时优先“读老写新”，必要时提供一次性迁移流程

## 9. 文件读写接口设计草案

### 9.1 设计目标
- 将文件系统细节与业务逻辑解耦
- 将 JSON / 二进制序列化细节封装在独立模块内
- 读写接口尽量稳定，便于未来替换底层格式
- 对损坏文件、缺失文件、部分写入失败提供可恢复策略

### 9.2 建议的数据对象
建议先定义一组纯数据对象，供采集、录制、UI、文件读写共用：
- `WorkspaceManifest`
- `DeviceInfo`
- `SessionMetadata`
- `SessionSummary`
- `MouseSample`
- `UiSnapshot`
- `SampleFileHeader`

### 9.3 核心接口建议

#### `IWorkspaceRepository`
建议接口：
- `bool openWorkspace(const QString& rootPath)`
- `bool createWorkspace(const QString& rootPath)`
- `WorkspaceManifest loadManifest() const`
- `bool saveManifest(const WorkspaceManifest& manifest)`
- `QVector<SessionMetadata> listSessions() const`
- `QVector<DeviceInfo> listDevices() const`
- `QString sessionPath(const QString& sessionUuid) const`

#### `ISessionStorage`
建议接口：
- `bool createSession(const SessionMetadata& meta)`
- `std::optional<SessionMetadata> loadSession(const QString& sessionUuid) const`
- `bool saveSession(const SessionMetadata& meta)`
- `std::optional<SessionSummary> loadSummary(const QString& sessionUuid) const`
- `bool saveSummary(const QString& sessionUuid, const SessionSummary& summary)`
- `bool removeSession(const QString& sessionUuid)`

#### `IDeviceStorage`
建议接口：
- `QVector<DeviceInfo> loadDevices() const`
- `bool saveDevices(const QVector<DeviceInfo>& devices)`
- `bool upsertDevice(const DeviceInfo& device)`
- `std::optional<DeviceInfo> findDevice(const QString& deviceId) const`

#### `ISampleBinaryWriter`
建议接口：
- `bool open(const QString& filePath, const SampleFileHeader& header)`
- `bool append(const MouseSample& sample)`
- `bool appendBatch(const QVector<MouseSample>& samples)`
- `bool flush()`
- `void close()`
- `qint64 bytesWritten() const`
- `quint64 sampleCount() const`

#### `ISampleBinaryReader`
建议接口：
- `bool open(const QString& filePath)`
- `std::optional<SampleFileHeader> readHeader() const`
- `std::optional<MouseSample> readNext()`
- `QVector<MouseSample> readRange(quint64 startIndex, quint64 count)`
- `bool seekToSample(quint64 sampleIndex)`
- `void close()`

### 9.4 具体类建议
- `WorkspaceRepository`
- `SessionStorage`
- `DeviceStorage`
- `SampleBinaryWriter`
- `SampleBinaryReader`
- `JsonCodec`

### 9.5 异常恢复策略建议
- `manifest.json` 缺失时可重建最小 manifest
- `summary.json` 缺失时将 session 标记为不完整
- `samples.bin` 尾部损坏时允许截断到最后一条完整记录
- JSON 文件采用临时文件 + 原子替换
- 单个会话目录损坏不影响整个 workspace 打开

# 鼠标实时数据 PC App 需求文档（V0.3）

## 1. 产品概述
开发一款 Windows PC 桌面应用，用于实时采集、展示、记录和分析鼠标相关数据，帮助用户观察鼠标性能表现、输入稳定性和设备状态。

当前定位为：**面向电竞用户、外设评测用户、厂商/测试人员的鼠标数据监控 + 性能分析工具**。

产品需支持**最小化后在后台持续录制数据**，并在 **UI 打开且处于活跃状态时才执行实时可视化**。

## 2. 背景
用户希望获得比普通系统鼠标设置更细粒度的数据，例如：
- 实时位置（Position）
- 移动增量（Delta X / Delta Y）
- 轮询率（Polling Rate / Hz）
- 滚轮事件
- 按键事件
- 鼠标硬件/专业指标（待进一步定义）

现有系统工具通常只能提供基础设置，无法满足鼠标性能观察、评测、调试、优化等需求，因此需要一个专门的桌面应用。

## 3. 产品目标
### 3.1 核心目标
- 实时稳定地获取鼠标输入数据
- 用低延迟界面展示关键指标
- 支持会话级记录与分析
- 支持结果导出，服务评测、验证和问题定位
- 为“专业数据”预留扩展能力

### 3.2 非目标（当前阶段）
- 不做鼠标驱动替代
- 不直接修改鼠标硬件参数
- 不做云端同步与账号体系
- 不做跨平台（当前先聚焦 Windows）

## 4. 目标用户
本产品当前面向以下用户：
- 电竞/高玩用户：关注轮询率稳定性、输入表现、轨迹质量
- 外设评测用户：关注不同鼠标、不同设置、不同场景下的数据表现
- 外设厂商/测试人员：关注调试、验证、回归对比、问题定位

当前版本**不以普通用户为主要目标用户**。

## 5. 典型使用场景
1. 用户打开应用后，实时查看当前鼠标位置、移动速度、轮询率波动。
2. 用户开始一次测试会话，移动鼠标并滚轮/点击，结束后查看统计数据。
3. 用户对不同鼠标、不同 USB 端口、不同系统设置下的数据进行对比。
4. 用户导出测试结果，用于评测、售后、调试或研发分析。

## 6. 核心功能需求

### 6.1 实时数据采集
系统需要支持实时采集以下基础数据：
- 鼠标当前位置（X, Y）
- 位移增量（Delta X, Delta Y）
- 时间戳
- 左/右/中键状态
- 侧键状态（如可获取）
- 滚轮滚动事件
- 原始输入事件计数
- 轮询率估算值（Hz）

### 6.2 实时监控面板
信息架构采用：**首页以监控仪表盘为主，同时通过左侧导航拆分页面**。

应用应提供实时监控面板，建议包含：
- 当前连接鼠标信息
- 实时坐标
- 当前轮询率
- 最近 1s / 5s / 30s 平均轮询率
- 最大 / 最小 / 波动情况
- 当前移动速度
- 点击/滚轮事件计数

### 6.3 数据可视化
建议提供以下展示方式：
- 轮询率实时曲线
- 鼠标移动轨迹图
- 速度变化曲线
- 点击事件时间线
- 滚轮事件时间线

其中关键指标展示要求如下：
- 轮询率稳定性：需支持平均值、最大值、最小值、标准差、稳定度评分、波动图
- 轨迹平滑度 / 抖动：需支持轨迹图、抖动数值、评分/等级

### 6.4 会话记录
用户可启动/停止一次测试会话。
会话记录内容建议包括：
- 开始/结束时间
- 设备信息
- 测试时长
- 原始采样数据
- 汇总统计结果
- 测试模式类型
- 关键评分结果

系统应支持历史会话保存，并支持多次测试结果的批量对比。

当应用窗口最小化或 UI 不活跃时，若用户已开启录制，系统仍需在后台持续采集和记录数据；此时暂停高频实时可视化，仅保留必要状态指示。用户重新打开并激活 UI 后，再恢复图表和实时数据可视化刷新。

### 6.5 数据导出
应用应支持导出测试结果。

第一版导出重点为：
- 原始采样数据

候选格式：
- CSV：便于表格分析
- JSON：便于程序处理

以下内容暂不作为第一版重点导出项：
- 图表截图
- 自动生成报告
- 对比结果专用导出

### 6.6 专业数据需求
当前已明确优先关注以下专业数据：
- 轮询率稳定度
- 轨迹平滑度 / 输入抖动（Jitter）
- 设备信息（VID / PID / 厂商 / 产品名 / 连接状态 / 电量 / 固件版本等，按可获取能力尽量完整展示）

后续候选扩展包括：
- 点击延迟
- DPI / CPI 估算
- 平滑/加速度特征
- 传感器异常点
- 抬起距离相关测试能力

## 7. 非功能需求
### 7.1 性能
- 实时展示刷新流畅，不明显掉帧
- 长时间采集时内存占用可控
- 对系统性能影响可接受
- 最小化或 UI 不活跃时，应降低渲染与图表刷新开销，优先保障后台录制稳定性

### 7.2 准确性
- 数据时间戳尽可能精确
- 轮询率计算方法需明确且可解释
- 不同采样模式下结果应可复现

### 7.3 稳定性
- 插拔鼠标后应用不崩溃
- 多鼠标场景下行为可预期
- 最小化/切后台后采集策略明确

### 7.4 易用性
- 默认界面即可查看核心指标
- 专业功能不影响普通用户上手
- 关键数据支持复制/导出

## 8. 技术约束与实现假设
- 平台：Windows
- 桌面框架：Qt
- 核心实现：C++ / STL / Windows SDK
- 本地存储：工作目录文件库（基于文件系统的会话目录、原始采样文件、摘要文件）
- 鼠标数据获取预计优先基于 Windows 原始输入能力（Raw Input / 相关系统接口）
- 若需更深层设备数据，后续可能需要厂商 SDK、驱动接口或 HID 层扩展

## 9. 风险与难点
- Windows 通用接口未必能获取所有“专业数据”
- 不同品牌/型号鼠标能力差异较大
- 高频采样下 UI 刷新与数据记录需要解耦
- 轮询率、点击延迟等指标的定义和计算方式需要标准化

## 10. MVP 建议范围
建议第一版（MVP）先做：
- 单机 Windows 桌面应用
- 仅监控当前活动鼠标
- 首页监控仪表盘
- 左侧导航分页面结构
- 区分“实时监控模式”和“测试模式”两种运行状态
- 首页直接展示当前测试状态与最近一次测试摘要
- 支持最小化后后台持续录制
- UI 仅在窗口打开且活跃时进行实时图表与数据可视化
- 实时数据采集与监控面板（位置 / Delta / 轮询率 / 按键 / 滚轮）
- 轮询率实时曲线
- 轮询率稳定性分析：平均值 / 最大值 / 最小值 / 标准差 / 稳定度评分 / 波动图
- 轨迹可视化
- 轨迹平滑度 / 抖动分析：轨迹图 / 抖动数值 / 评分等级
- 会话记录与历史保存
- 多次测试结果批量对比
- 对比维度优先支持：轮询率稳定度 / 波动、轨迹图、峰值/最低值、设备信息差异
- 原始采样数据导出（CSV / JSON）
- 测试模式：手动开始 / 手动停止，不预设固定时长或标准流程
- 无法识别的设备字段显示为 N/A
- 先基于 Windows 通用接口实现，并为后续 HID / 厂商 SDK 扩展预留能力

暂不纳入 MVP：
- 多鼠标同时监控
- 点击延迟测试
- 图表截图导出
- 自动生成专业报告
- 对比结果专用导出
- 预设测试流程模板
- 云端能力
- 复杂评测流程编排
- 厂商私有协议深度接入
- 普通用户导向的轻量模式

## 11. 验收标准（初稿）
- 能稳定识别当前活动鼠标输入并实时显示基础数据
- 能输出轮询率实时值、区间统计值，以及平均值、最大值、最小值、标准差、稳定度评分和波动图
- 能输出轨迹图、抖动数值以及轨迹平滑度评分/等级
- 能完成轮询率测试、轨迹/稳定性测试，并生成可保存结果
- 能记录一次完整测试会话、保存历史，并支持多次测试结果批量对比
- 首页能显示当前测试状态与最近一次测试摘要
- 应用最小化后，若录制已开启，仍能持续后台采集和记录数据
- UI 恢复活跃后，能继续正常展示实时图表和可视化结果
- 能导出原始采样数据
- 常见操作（移动、点击、滚轮）均能被正确记录
- 无法识别的设备字段以 N/A 显示
- 断开/切换鼠标后应用不崩溃

## 12. 已确认方向
- 目标用户：电竞/高玩用户、外设评测用户、厂商/测试人员
- 核心价值：实时监控 + 录制分析 + 结果导出
- 信息架构：首页采用监控仪表盘，整体采用左侧导航分页面结构
- 运行模式：区分“实时监控模式”和“测试模式”
- 首页摘要：直接展示当前测试状态与最近一次测试摘要
- 录制策略：支持最小化后后台持续录制
- 可视化策略：仅在 UI 打开且处于活跃状态时进行实时图表与数据可视化
- 专业数据优先级：轮询率稳定性、轨迹平滑度/抖动、设备信息
- 轮询率稳定性表达：平均值、最大值、最小值、标准差、稳定度评分、波动图全量提供
- 轨迹平滑度 / 抖动表达：轨迹图 + 抖动数值 + 评分/等级
- 点击延迟：用户关注，但当前不纳入第一版 MVP
- 测试模式优先级：轮询率测试、轨迹/稳定性测试
- 测试交互：手动开始 / 手动停止，不预设固定时长或标准模板
- 多鼠标策略：第一版仅关注当前活动鼠标
- 数据留存：保存历史会话，并支持多次测试结果批量对比
- 对比重点：轮询率稳定度/波动、轨迹图、峰值/最低值、设备信息差异
- 导出重点：原始采样数据导出
- 设备信息粒度：尽可能完整，包含固件等信息（以接口能力为准）
- 无法识别字段：显示为 N/A
- 技术路径：第一版优先基于 Windows 通用接口，后续可扩展 HID / 厂商 SDK

## 13. 技术方案初稿

### 13.1 总体架构
建议采用“采集层 / 计算层 / 存储层 / 展示层”四层结构：

- 采集层：负责从 Windows 原始输入接口持续接收鼠标事件
- 计算层：负责轮询率、稳定度、轨迹抖动、统计汇总等指标计算
- 存储层：负责会话、原始采样数据、统计结果、设备信息落盘
- 展示层：负责 Qt UI 页面、图表、状态面板、历史对比展示

核心原则：**采集与录制不能依赖 UI，展示层只消费数据，不驱动数据产生**。

### 13.2 线程与模块划分建议
建议至少拆分以下职责：

1. UI 主线程
- 负责窗口、页面切换、图表刷新、用户操作响应
- 不直接承担高频采样计算与磁盘写入

2. 输入采集线程
- 负责接收 Raw Input / Windows 鼠标输入事件
- 将事件标准化为统一采样结构
- 推送到内存缓冲区或事件队列

3. 统计计算线程
- 消费采样事件
- 计算实时指标、窗口统计值、评分结果
- 生成供 UI 使用的轻量快照数据

4. 录制/存储线程
- 将原始采样数据与测试会话数据异步写入 SQLite
- 负责批量写入、刷盘节流、会话结束汇总

可复用现有 [LambdaThread.h](LambdaThread.h) 作为后台线程承载基础，但后续建议逐步明确独立模块边界。

### 13.3 数据流设计
建议数据流如下：

1. Windows 输入事件进入采集层
2. 采集层生成标准化 MouseSample
3. MouseSample 同时进入两条链路：
   - 实时计算链路：用于当前轮询率、轨迹、抖动、状态摘要
   - 录制链路：用于原始数据持久化
4. 计算层按固定频率输出 UiSnapshot
5. UI 活跃时消费 UiSnapshot 并刷新图表
6. UI 最小化或不活跃时，暂停高频图表刷新，但采集链路和录制链路继续运行
7. 测试结束后生成 SessionSummary，用于历史记录与对比页面

### 13.4 后台录制与前台可视化策略
建议明确区分两套状态：

#### 应用可见性状态
- 前台活跃
- 最小化
- 后台非活跃

#### 业务运行状态
- 空闲
- 实时监控中
- 测试录制中
- 测试结束待查看

处理原则：
- 只要业务状态处于“测试录制中”，即使窗口最小化，也必须持续采集与写盘
- 前台活跃时，允许高频刷新图表和实时数值
- 最小化或后台非活跃时，暂停非必要图表刷新，只保留轻量状态更新
- 恢复前台后，UI 从最新快照恢复，不要求重放全部历史帧

### 13.5 内存缓冲与节流建议
为避免高频鼠标事件直接压垮 UI 或数据库，建议：

- 采集层使用环形缓冲区或无锁/低锁队列暂存事件
- 存储层采用批量写入，不对每个事件单独提交事务
- UI 刷新频率与采样频率解耦，例如 UI 每秒刷新固定次数
- 图表展示使用时间窗口数据，而非无限累积点数
- 长时间录制时，对内存中的可视化缓存做滚动裁剪

### 13.6 关键数据结构建议
建议至少抽象以下数据结构：

#### MouseSample
单条原始输入采样，建议字段包括：
- timestamp_us
- device_id
- pos_x
- pos_y
- delta_x
- delta_y
- wheel_delta
- button_flags
- event_type
- raw_event_index

#### UiSnapshot
供 UI 刷新的轻量快照，建议字段包括：
- current_hz
- avg_hz_1s
- avg_hz_5s
- avg_hz_30s
- hz_min
- hz_max
- hz_stddev
- stability_score
- current_speed
- jitter_value
- smoothness_score
- latest_device_info
- latest_test_status

#### SessionSummary
一次测试的汇总结果，建议字段包括：
- session_id
- test_mode
- start_time
- end_time
- duration_ms
- avg_hz
- hz_stddev
- hz_stability_score
- jitter_value
- smoothness_score
- sample_count
- device_snapshot

### 13.7 SQLite 数据模型初稿
建议最少包含以下表：

1. sessions
- 保存测试会话基础信息
- 字段示例：id, test_mode, start_time, end_time, duration_ms, device_id, summary_json

2. mouse_samples
- 保存原始采样数据
- 字段示例：id, session_id, timestamp_us, pos_x, pos_y, delta_x, delta_y, wheel_delta, button_flags, event_type, raw_event_index

3. devices
- 保存设备识别信息
- 字段示例：id, device_id, vendor_id, product_id, manufacturer, product_name, firmware_version, connection_type, battery_level, extra_json

4. session_metrics
- 保存会话级统计指标
- 字段示例：id, session_id, avg_hz, min_hz, max_hz, hz_stddev, stability_score, jitter_value, smoothness_score

后续如需复杂对比，可增加视图或派生表，而不是过早增加大量冗余表。

### 13.8 页面与数据关系建议
- 首页：读取 UiSnapshot 和当前状态摘要
- 测试页：控制测试开始/停止，并展示测试过程中的实时统计
- 历史页：读取 sessions 列表和 session_metrics 汇总
- 对比页：按选中的多个 session 拉取 summary 与关键图表数据
- 设备页：读取 devices 信息，无法识别字段显示 N/A

### 13.9 指标计算实现建议
第一版建议采用可解释、易落地的计算方式：

- 轮询率：基于相邻输入事件时间差估算瞬时 Hz，再按窗口求均值、最大值、最小值、标准差
- 稳定度评分：基于目标轮询率附近波动程度做归一化评分
- 抖动值：基于轨迹局部偏移、短时波动或离散程度计算
- 平滑度评分：由抖动值反向映射得到评分

注意：第一版先保证“可解释、可复现”，不要一开始引入过于黑盒的评分算法。

### 13.10 MVP 开发优先级建议
建议按以下顺序开发：

1. 输入采集打通
- 能稳定接收鼠标原始事件
- 能识别当前活动鼠标

2. 后台录制链路
- 最小化后持续采集
- 原始数据写入 SQLite

3. 实时监控 UI
- 首页指标面板
- 基础状态栏
- 前台活跃时刷新图表

4. 测试模式
- 手动开始 / 停止
- 生成 SessionSummary

5. 历史与对比
- 历史记录列表
- 多次结果批量对比

6. 设备信息增强
- 完善 VID / PID / 厂商 / 固件 / 电量等字段获取

## 14. 数据库 Schema 草案

### 14.1 设计原则
- 原始采样与汇总结果分离存储
- 高频写入表保持扁平，避免过度关联
- 汇总指标单独存储，方便历史列表与对比查询
- 设备信息尽量单独归档，避免每条采样重复写入大量冗余字段
- 第一版优先保证可写入、可查询、可导出，不做过度复杂的范式设计

### 14.2 表结构建议

#### 表：devices
用途：保存识别到的鼠标设备信息。

建议字段：
- id INTEGER PRIMARY KEY
- device_id TEXT NOT NULL UNIQUE
- vendor_id TEXT
- product_id TEXT
- manufacturer TEXT
- product_name TEXT
- serial_number TEXT
- firmware_version TEXT
- connection_type TEXT
- battery_level INTEGER
- extra_json TEXT
- first_seen_at TEXT NOT NULL
- last_seen_at TEXT NOT NULL

说明：
- `device_id` 用于标识系统识别到的设备实例
- 无法识别的字段允许为空，UI 层统一显示为 N/A
- 非标准或厂商扩展字段放入 `extra_json`

#### 表：sessions
用途：保存每次测试或录制会话的元信息。

建议字段：
- id INTEGER PRIMARY KEY
- session_uuid TEXT NOT NULL UNIQUE
- device_id TEXT
- test_mode TEXT NOT NULL
- run_mode TEXT NOT NULL
- start_time TEXT NOT NULL
- end_time TEXT
- duration_ms INTEGER
- sample_count INTEGER DEFAULT 0
- is_background_recording INTEGER NOT NULL DEFAULT 0
- status TEXT NOT NULL
- notes TEXT
- created_at TEXT NOT NULL
- updated_at TEXT NOT NULL

说明：
- `test_mode` 可表示 polling / trajectory / monitor 等模式
- `run_mode` 可表示 monitor / testing
- `status` 可表示 running / finished / aborted
- 最小化后台录制时，`is_background_recording` 记录该会话行为特征

#### 表：mouse_samples
用途：保存高频原始鼠标采样。

建议字段：
- id INTEGER PRIMARY KEY
- session_id INTEGER NOT NULL
- timestamp_us INTEGER NOT NULL
- event_index INTEGER NOT NULL
- pos_x INTEGER
- pos_y INTEGER
- delta_x INTEGER NOT NULL
- delta_y INTEGER NOT NULL
- wheel_delta INTEGER DEFAULT 0
- button_flags INTEGER NOT NULL DEFAULT 0
- event_type TEXT NOT NULL
- instantaneous_hz REAL
- extra_json TEXT

建议索引：
- INDEX idx_mouse_samples_session_time(session_id, timestamp_us)
- INDEX idx_mouse_samples_session_event(session_id, event_index)

说明：
- `event_type` 可表示 move / button / wheel / combined
- `instantaneous_hz` 可选存储，用于导出和调试；也可运行时计算

#### 表：session_metrics
用途：保存会话级汇总统计。

建议字段：
- id INTEGER PRIMARY KEY
- session_id INTEGER NOT NULL UNIQUE
- avg_hz REAL
- min_hz REAL
- max_hz REAL
- hz_stddev REAL
- stability_score REAL
- jitter_value REAL
- smoothness_score REAL
- peak_speed REAL
- avg_speed REAL
- total_distance REAL
- left_click_count INTEGER DEFAULT 0
- right_click_count INTEGER DEFAULT 0
- middle_click_count INTEGER DEFAULT 0
- wheel_event_count INTEGER DEFAULT 0
- metric_version TEXT
- summary_json TEXT

说明：
- 固定列保存高频查询字段
- 更灵活的派生结果写入 `summary_json`
- `metric_version` 便于后续升级算法后做兼容和回溯

#### 表：ui_state_snapshots（可选）
用途：保存最近一次 UI 展示状态或上次测试摘要。

建议字段：
- id INTEGER PRIMARY KEY
- snapshot_type TEXT NOT NULL
- payload_json TEXT NOT NULL
- updated_at TEXT NOT NULL

说明：
- 第一版可不落盘，先以内存维护
- 如果希望启动后立即恢复最近状态，可后续启用此表

### 14.3 查询场景建议
- 首页摘要：读取当前运行状态 + 最近一个已完成 session 的 `session_metrics`
- 历史记录页：查询 `sessions` 并左联 `session_metrics`
- 对比页：按多个 `session_id` 批量读取 `session_metrics`
- 原始数据导出：按 `session_id` 查询 `mouse_samples`
- 设备信息页：读取 `devices`

### 14.4 数据归档与容量建议
- 长时间录制时，`mouse_samples` 可能增长很快
- 第一版先不做复杂分表，但需预留按 session 清理数据的能力
- 导出完成后可允许用户手动删除历史 session
- 后续如需长时后台录制，可考虑按会话分库或按日期归档

## 15. C++ / Qt 模块设计草案

### 15.1 模块划分
建议按目录和职责拆分为以下模块：

1. app
- 应用启动、全局生命周期管理
- 管理模式切换、全局状态、依赖装配

2. input
- Windows Raw Input 接入
- 鼠标设备枚举与事件标准化

3. pipeline
- 事件队列、缓冲、分发
- 解耦采集、计算、存储、UI

4. metrics
- 轮询率、稳定度、速度、抖动、平滑度计算
- 输出实时统计与会话汇总

5. recorder
- 会话开始/停止
- 原始数据批量写入 SQLite
- 会话结束汇总落盘

6. repository
- SQLite 访问封装
- sessions / devices / metrics / samples 查询写入

7. ui
- 主窗口、页面、图表、状态栏
- 消费 UiSnapshot，不直接操作底层采样数据

### 15.2 核心类建议

#### `AppController`
职责：
- 作为应用总协调器
- 初始化采集、计算、录制、仓储、UI
- 管理实时监控模式 / 测试模式切换
- 响应窗口活跃状态变化

#### `MouseInputSource`
职责：
- 封装 Raw Input 注册与事件接收
- 输出标准化 `MouseSample`
- 屏蔽 Windows 原始消息细节

#### `MouseSample`
职责：
- 作为统一事件模型
- 在采集、计算、存储之间传递

#### `SampleDispatcher`
职责：
- 将 `MouseSample` 分发到计算链路和录制链路
- 作为采集层与后续模块之间的解耦边界

#### `RealtimeMetricsEngine`
职责：
- 维护滑动窗口
- 计算当前 Hz、均值、标准差、抖动值、速度等实时指标
- 周期性生成 `UiSnapshot`

#### `SessionRecorder`
职责：
- 管理录制会话生命周期
- 控制原始采样批量写入
- 生成 `SessionSummary`

#### `SessionRepository`
职责：
- 提供会话、指标、原始采样、设备信息的数据库读写接口
- 隐藏 SQLite 细节

#### `DeviceInfoService`
职责：
- 负责设备识别、字段补全
- 聚合 VID / PID / 厂商 / 固件 / 电量等信息
- 对取不到的字段返回空值，由 UI 统一显示 N/A

#### `UiStateStore`
职责：
- 保存当前 UI 所需的最新轻量状态
- 提供首页摘要、最近测试结果、当前运行状态
- 避免 UI 直接访问后台线程内部数据结构

### 15.3 Qt 页面建议
建议页面与类对应如下：

- `MainWindow`：整体容器、左侧导航、状态栏
- `DashboardPage`：实时监控首页
- `TestPage`：测试模式操作页
- `HistoryPage`：历史记录列表页
- `ComparePage`：多次结果对比页
- `DevicePage`：设备信息页

### 15.4 线程协作建议
- `MouseInputSource` 在独立线程接收和标准化输入
- `RealtimeMetricsEngine` 在后台线程计算实时指标
- `SessionRecorder` 在后台线程批量写库
- UI 主线程只接收 `UiSnapshot` 和查询结果
- 线程之间优先通过 Qt signal/slot 或明确的线程安全队列通信

### 15.5 状态机建议
建议建立统一状态机，避免页面逻辑分散：

#### 应用状态
- AppStarting
- AppIdle
- AppMonitoring
- AppTesting
- AppMinimizedRecording
- AppViewingHistory

#### 会话状态
- SessionNone
- SessionPreparing
- SessionRunning
- SessionStopping
- SessionFinished
- SessionAborted

这样可以明确：
- 最小化时是否继续录制
- 恢复前台时 UI 如何刷新
- 会话结束后摘要如何生成和展示

### 15.6 与当前仓库结构的衔接建议
结合当前仓库现状：
- [main.cpp](main.cpp) 继续承担应用启动与主窗口初始化
- [Gg.h](Gg.h) / [Gg.cpp](Gg.cpp) 适合逐步演进为 `MainWindow` 主容器
- [Gg.ui](Gg.ui) 适合承载左侧导航和主内容区骨架
- [LambdaThread.h](LambdaThread.h) 可继续作为后台线程封装的过渡方案

第一阶段不必一次性重构成大量文件，可以先按模块边界组织类，再逐步拆分实现文件。
